## New findings this round

1. **Check-mode/fresh-host ordering gaps in `freeipa-nfs-server-apply.yml` and
   `freeipa-nfs-client-apply.yml`** (playbook defects, fixed, see diff above): on a genuinely fresh
   host, `freeipa-client-apply.yml`'s real enrollment is correctly gated `when: not
   ansible_check_mode`, but several downstream gates/tasks in the NFS server/client playbooks
   asserted on that not-yet-real state unconditionally, so a full-site `--check --diff` preview on
   a fresh host always failed. This is the same class of gap previously fixed once for
   `docker_container`/`docker_image` tasks; it was not previously fixed for these NFS gates.
2. **`ipa service-add` requires `--force` in a no-DNS-managed topology** (playbook defect, fixed):
   this codebase's FreeIPA design deliberately has no DNS zone management (`--no-host-dns`, static
   `/etc/hosts` pinning). `ipa host-add` already works around this via `--ip-address`; `ipa
   service-add` had no equivalent and failed with "does not have corresponding DNS A/AAAA record"
   on the real apply. Fixed by adding `--force`, FreeIPA's documented equivalent for services.
3. **Implicit ordering dependency: NFS share group ownership vs. identity reconcile** (documentation
   finding, no code change): `freeipa-nfs-server-apply.yml`'s real `chgrp` to a roster-managed POSIX
   group (e.g. `data-project-alpha-rw`) requires that group to already exist and be resolvable via
   SSSD — which only happens after `pilot reconcile` (`freeipa-identity`) has run at least once. The
   runbook's documented order (§3.4 site-wide deploy, then §3.5 identity reconcile) does not surface
   this dependency; on a genuinely fresh host with an NFS share whose ownership group is
   roster-managed, the identity reconcile must run first (or the site-wide deploy re-run afterward).
4. **`freeipa_roster_file` must be set on every host whose playbook reads it, not just
   `freeipa-server`** (documentation gap, closed this round): `freeipa-nfs-server-apply.yml` (which
   runs on `nexus` in this topology) also does `include_vars: file: "{{ freeipa_roster_file }}"` —
   the runbook's Prerequisites previously only said to set this on "the managed host" ambiguously.
5. **Roster authoring conventions confirmed by validation gates** (not a defect — a roster-authoring
   note worth documenting): a roster group's `category` must match its name's prefix
   (`team-`/`data-`/`access-`/`role-`), and `hbac.rules[].subjects.groups` may only reference
   `category: access` groups while `sudo.rules[].subjects.groups` may only reference `category:
   role` groups — a single group cannot serve both purposes directly.
6. **Brand-new roster users need the `kinit` forced-password-change dance regardless of
   `force_change: false`**: the existing runbook gotcha frames this around `force_password: true` or
   an out-of-band reset; this round confirmed it also applies to a genuinely first-time onboard with
   `force_change: false` in the roster — FreeIPA's own `ipa passwd` always arms the forced-change
   flag on first assignment, independent of the roster's flag (which only controls whether that
   *repeat* applies on steady-state reruns).
7. **`pilot deploy`'s topology graph prints resource-minimum recommendations that exceed this
   runbook's own VM sizing** (informational only): the wizard reported `client-vm` needing `≥ 6
   vCPU / 4 GiB RAM / 29 GiB` and `freeipa-server` needing `≥ 6 vCPU / 6 GiB RAM / 30 GiB`, both
   above the runbook's provisioned sizing (2 vCPU/2 GiB/20 GiB and 2 vCPU/4 GiB/30 GiB
   respectively). The apply completed successfully at the smaller sizing in both round 12 and this
   round, so the wizard's estimate is conservative/additive rather than a hard requirement — noted
   for awareness, not acted on.

## Process note

Two secret-exposure incidents occurred during vault authoring (both disposable sandbox credentials
generated solely for this run's throwaway VMs, never reused, VMs torn down at teardown): once when
a follow-up TREC script displayed already-set vault values on screen without `--secret-env`
declared for that specific invocation, and once when an external process timeout killed a `trec
drive` run mid-step, whose failure diagnostic bypassed redaction. Both casts were quarantined
(never published) and the affected secrets were regenerated. Corrected practice going forward:
declare `--secret-env` for every script that touches the vault screen, including read-only
navigation passes, and never race an external timeout against a vault-touching `trec drive`
invocation.
