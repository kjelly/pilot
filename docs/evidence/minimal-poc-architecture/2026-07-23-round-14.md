## Real bug found + fixed (Go source, authorized)

`pilot edit --actions` could not build a `hosts.yml` with more than one host.
`createHost`'s `ensureHostsList` handled returning from the top menu or an already-open host list,
but not from a *different host's own menu* (exactly where every prior host's `enable_role`/
`set_host_field` sequence leaves the router) — it returned a hard error instead of first choosing
"↩ 返回主機清單" the way its sibling `ensureHostMenu` already did. Confirmed via cast + trace
(`casts/01-edit-hosts-actions.cast` first attempt, `evidence/edit-hosts-trace.jsonl`): no `hosts.yml`
was ever written before the error. Every existing test for this feature only ever drove a single
host (`web-1`), so the multi-host path had never been exercised.

Fix (authorized by the user before editing): make `ensureHostsList` return via
`d.choose(r, "返回主機清單")` for that case, mirroring `ensureHostMenu`. Added
`TestEditAutomationDriverMultiHostFlow` (two hosts, two roles, save, parse back) as a regression
test. `go test ./cmd/pilot/cmd -count=1` (276 tests) and the focused driver tests pass.

```diff
diff --git a/cmd/pilot/cmd/edit_automation_driver.go b/cmd/pilot/cmd/edit_automation_driver.go
@@ -189,7 +189,7 @@ func (d *automationDriver) ensureHostsList(r *editRouterModel) error {
 		case strings.Contains(list.title, "編輯") && strings.Contains(list.title, "選一台主機"):
 			return nil
 		case strings.Contains(list.title, "選要編輯的項目"):
-			return fmt.Errorf("host menu must return to host list first")
+			return d.choose(r, "返回主機清單")
 		}
```

Full diff (including the new test) is in the working tree at the commit/tree recorded above; not
yet committed as of this writing.

## Documentation-only finding (no code fix)

`freeipa-identity-apply.yml`'s own top-of-file comment ("The roster file … MUST also define
`ipa_admin_password`") and `contracts/freeipa-identity.yaml`'s `groupVars: [{name:
ipa_admin_password, required: true}]` both imply the roster file itself should carry a bare
top-level `ipa_admin_password` key. It cannot: the same playbook's canonical-schema gate
(`freeipa-identity-apply.yml:115-130`) explicitly rejects any top-level roster key outside a fixed
allowlist that does not include `ipa_admin_password` — confirmed live (preview failed, no
mutation, `casts/06-reconcile-identity.cast` first attempt). The two requirements are mutually
exclusive for any canonical (`schema_version: 1`) roster. The actually-working path — already
correctly documented in this runbook's §3.5 — is to answer **yes** to the "偵測到
`.vault/main.yaml`" vars-file prompt (supplies the bare `ipa_admin_password` the *contract*
preflight check wants) while the roster file itself only ever carries `freeipa.admin.password`
(what the canonical Ansible code path actually reads). This round's first reconcile attempt failed
because the agent misread that instruction and pointed the vars-file prompt at the roster instead;
the fix was purely a scenario/roster-authoring correction, not a product change. Classified as a
stale/misleading comment + contract declaration, not a blocking implementation defect, since a
working path already exists and is already documented correctly elsewhere in this runbook.

## `pilot actions`/`pilot services` usage summary (what the user asked to verify)

- `pilot services status` showed the host-local cache already running (`profile=dev-lite`) from an
  earlier session; reused as-is per the delivery-test skill's own guidance (it is host-level cache
  state, not part of the disposable candidate/workspace). All three `vm-target up` calls used
  `--services local`.
- `pilot actions list` / `pilot actions schema` were read fresh from the freshly-built binary before
  writing any scenario (per this repo's own `agent-run-test.md` instruction) — 6 actions:
  `create_host`, `set_host_field`, `enable_role`, `save_hosts`, `deploy`, `reconcile`.
- `pilot edit --actions` built the entire 3-host, 22-role-assignment `hosts.yml` in one
  non-interactive run (after the fix above) — a materially more robust replacement for
  `trec drive`'s `DOWN <n>` role-checklist counting for this specific screen, since the driver
  resolves against the live in-memory model, not rendered terminal text.
- `pilot deploy --actions` and `pilot reconcile --actions` both required a real PTY despite taking
  no live keystrokes (same TTY guard as the interactive path) — wrapped in a plain `trec` recorder
  (no `trec drive` script needed, since the automation drives itself from the scenario file).
- group_vars and `.vault/` editing are **not yet covered** by semantic actions (schema only covers
  `hosts.yml` fields) — still require interactive `pilot edit` (trec MCP or `trec drive`).
- Every `deploy`/`reconcile` prompt-answer scenario required enumerating the exact prompt text from
  `deploy.go` source (same "recompute fresh, don't trust memory" discipline as index-based
  `trec drive` scripts) — including a real trap: the same literal confirm prompt
  ("確定要執行以上指令嗎？") is asked twice (once for preview, once for apply) in one deploy/reconcile
  run, and the scenario validator rejects two answers with an identical `prompt` string. Fixed by
  giving the two answers slightly different (but still-matching-by-substring) prompt text — see
  `scripts/deploy-site-scenario.json`.
