## Why this round exists

The runbook's §3.2 built its three disposable VMs with three hand-typed `pilot vm-target up`
invocations. A declarative topology spec for this exact 3-node scenario
(`docs/topologies/minimal-poc-topology.yaml`) already existed — built for an earlier
`vm-target topology test` smoke run (2026-07-18 delivery-hardening work) — but the runbook's own
rebuild procedure had never been switched over to reference it. The user asked to evaluate whether
`vm-target topology` was the better fit for that step and, if so, rewrite the runbook and test it
for real. It is: the spec already captured the exact sizing/role facts §3.2's prose separately
hand-maintained (and could drift from), `topology up`/`down`/`status` scope themselves to exactly
the three spec-declared nodes (matching this runbook's own "resolve only the three exact target
domains" discipline more directly than a bare `vm-target list`), and `topology up` now provisions
concurrently and safely (the 2026-07-06 cross-process bring-up race this runbook used to cite as the
reason for serial `vm-target up` calls is long fixed).

## Real findings this round

### 1. Own roster-authoring gap: missing `nfs.servers` entry (caught by `--check`, zero mutation)

The hand-authored FreeIPA identity roster for this round initially omitted the top-level `nfs:` key
entirely (reasoned, incorrectly, that it was optional since a *share*'s ownership group is allowed
to be missing on a fresh host — see `freeipa-nfs-server-apply.yml`'s own self-healing comment).
`freeipa-nfs-server-apply.yml:79` unconditionally evaluates
`freeipa_roster.nfs.servers | selectattr(...)`, which raised a hard Jinja2 "object of type 'dict' has
no attribute 'nfs'" error during the very first `--check --diff` preview — before any host was
mutated. Fixed by adding one `nfs.servers` entry for `nexus.ipa.pilot.internal` with a valid
`service_principal` and `shares: []` (empty shares are handled gracefully; the server entry itself is
required). This is a scenario/roster-authoring gap, not a product defect — the runbook's own §2
already says to keep "Nexus's canonical FQDN/IP, NFS service principal, export, ACL, and automount
objects in that roster"; this round's first attempt simply didn't follow that fully.

### 2. Own inventory-authoring gap: missing `host_vars/nexus.yml` for `prometheus_site_label`

The site-wide deploy's `--check --diff` preview passed cleanly, but the real apply failed
`prometheus-apply.yml`'s "Gate: prometheus_site_label required" assertion (no default by design —
see `docs/verification/prometheus.md` §1.5). This variable is genuinely per-site, not one of the
cross-role host *addresses* the deploy wizard auto-detects, and `pilot edit` has no `host_vars/`
editor. Fixed by hand-writing `host_vars/nexus.yml` with `prometheus_site_label: site-nexus` (the
same tool-endorsed hand-edit exception as the vault's nested-YAML case) and re-running. Now
documented in the runbook's §3.3.

### 3. Operational gotcha (not a defect): transient FreeIPA LDAP enrollment race under concurrent apply

The first full-site real-apply attempt (after both roster/inventory gaps above were fixed and the
preview passed clean) failed `nexus`'s `freeipa-client` component:
`Joining realm failed: Operations error: Error checking for attribute uniqueness`, while `client-vm`'s
own concurrent enrollment (same play, `ANSIBLE_FORKS=20`) succeeded. Ansible's default behavior
excludes a host that fails a task from every subsequent play in the same run, which cascaded into
`nexus` never receiving `docker`/`wazuh-manager`/`seaweedfs-s3`/etc. that run, and `wazuh-fim`'s
agent-auth against `wazuh-manager` (only ever installed on `nexus`) then failed on the two other
hosts for an unrelated-looking reason (`agent-auth: ERROR: Unable to connect to enrollment service`).
Root-caused via `/var/log/ipaclient-install.log` (`ERROR Joining realm failed: ... attribute
uniqueness`) — a known 389-ds directory-server contention pattern when two clients join at nearly
the same moment. A plain re-run of `pilot deploy --actions` with the identical scenario succeeded
completely (`failed=0` all three hosts): `client-vm`'s enrollment was already durable, so only
`nexus` needed to (re-)join, and it no longer raced. This is independent of the `vm-target topology`
switch (Ansible's own per-play concurrency, not VM bring-up, is what raced) but is a real, previously
undocumented operational gotcha for this topology — added to the runbook's §6.

### 4. Pre-existing security-relevant gap — found, reported, fixed (authorized)

While reviewing this round's own evidence for secret leakage (see below), the `04-reconcile-identity.cast`
recording was found to contain the roster's plaintext initial password for user `alice` in cleartext,
printed by `freeipa-identity-apply.yml`'s own "Gate: canonical user objects are structurally valid"
assert task (`playbooks/apply/freeipa-identity-apply.yml`, ~line 145). That task loops over
`freeipa_roster.users` (each item includes `password.initial`) without `no_log: true` — unlike other
tasks in the same playbook (e.g. "Normalize canonical FreeIPA settings") which do carry
`no_log: true`. This meant **any** real run of this playbook (not just this sandbox) printed every
roster user's plaintext initial password to the `ansible-playbook` transcript/log by default, with no
verbosity flag required. This was outside this round's requested task (the `vm-target topology`
evaluation), so it was reported to the user rather than fixed silently; the user then authorized the
fix. Fixed by adding `no_log: true` to that task, matching the placement convention of the file's
other 7 `no_log` tasks. `ansible-playbook --syntax-check` and `make playbook-lint` both pass clean
afterward; no Go test references the task by name. The specific leaked value (alice's very first
bootstrap password) was rotated twice more within this same round before the evidence was kept, so it
was never a live credential by the time this evidence was published.

## Evidence integrity

All 6 casts produced this round passed `cast_verify` (`mcp__trec__cast_verify`): complete, exit 0,
0 secret-scan findings, safe to share — but not on the first attempt. An intermediate ad-hoc
verification script (§4 spot-check) used `set -x` and `sshpass -p "$PASSWORD"`, which echoed alice's
(then-current, test-only) password into the recorded `.cast` via the shell trace — `cast_verify`
caught this immediately (`4 likely unredacted secret(s)`, rule `sshpass-password`). The cast was
deleted, alice's password was rotated (twice, since the first rotation attempt itself failed for an
unrelated reason — see below), and the check was redone using `sshpass -e`/`SSHPASS` with no shell
tracing before being kept as evidence. Separately, an interactive `ssh ... "passwd"` attempt (no PTY,
piped heredoc) left alice's account in an inconsistent state (`pam_pwquality: user aborted password
change`, followed by unexplained subsequent auth failures with the presumed-still-valid password) —
recovered via the same admin-forced-reset + `kinit`-driven change pattern already proven earlier in
this round, rather than by guessing further at the interactive-`passwd`-over-`ssh` path. No `trec mcp`
sessions were left open.

## Follow-up: formalizing §4.1/§4.2 as a script

After this round closed, the user asked to evaluate converting the runbook's §4 verification items
(e.g. log-collection and SSH-login checks) into scripts. Weighed:

- **For:** repeatability/diffability (the ad-hoc `verify-section4-spotcheck.sh` used above already
  proved this out this round), faster re-runs across future rounds, and removing a class of human
  error — this same round's own evidence-integrity incident (a `set -x` + `sshpass -p` combination
  leaking alice's password into a `.cast`, see above) is exactly the kind of mistake a fixed,
  reviewed script eliminates at the source.
- **Against:** §3's build/deploy/reconcile steps exist to verify the live wizard/TUI flow itself —
  scripting those would replace the thing being tested with its assumed end state. A script can also
  silently drift from the environment (e.g. a hardcoded IP, or the port-number drift already caught
  and fixed in the 2026-07-17 runbook-consolidation round) and get trusted as "passing" without
  anyone noticing.

Resolution: only §4.1/§4.2's read-only assertions were formalized into a committed script,
[`scripts/minimal-poc-section4-spotcheck.sh`](../../../scripts/minimal-poc-section4-spotcheck.sh)
(JSON-line output per check, matching this repo's `scripts/verify-hello-localhost.sh` convention).
Unlike the ad-hoc version used above, it resolves `nexus`'s IP live from `pilot vm-target topology
status` instead of hardcoding `192.168.122.4` — the value that happened to work above was a static
DHCP reservation for this one bring-up, not a guaranteed constant. §3's deploy/reconcile steps and
§4.3/§4.4 remain manual/TREC-driven; see the runbook's own note at the top of §4.

Verification performed for this follow-up (scoped, not a full re-run): `bash -n` and `shellcheck`
both clean; live-tested the script's two genuinely new code paths — `pilot vm-target topology up`
→ `topology status`-based IP resolution (correctly resolved `nexus` to `192.168.122.4`) and `pilot
vm-target exec` connectivity to both `freeipa-server` and `nexus` — then torn back down
(`topology down`). The FreeIPA/SSH/Thanos assertion logic itself reuses the exact commands already
live-verified in this round's own `casts/05-verify-section4-spotcheck.cast`; a full site
redeploy was not repeated solely to re-prove that unchanged logic.
