# Worked example: computing indices for a `trec` script

This walks through turning `deploy_catalog.go` / `contracts.go` into a
correct `trec drive` script, using the exact class of bug that derails
these scripts most often: miscounting the total item list.

## 1. `pilot deploy`'s single-component menu

```bash
grep -n 'Key:' cmd/pilot/cmd/deploy_catalog.go
```

Each `Key:` line is one menu row, in file order, 0-indexed. Example
output shape (illustrative — re-run this yourself, don't trust these
numbers):

```
41:  Key: "core-infra-provider", ...   # index 0
47:  Key: "freeipa-server", ...        # index 1
52:  Key: "freeipa-client", ...        # index 2
56:  Key: "freeipa-identity", ...      # index 3
...
```

To select `freeipa-identity` from the top of the menu (cursor starts at
0): `DOWN 3` then `ENTER`.

## 2. `pilot edit`'s role checklist

```bash
grep -n 'Name:' internal/inventory/contracts.go
```

Same idea — the Nth `Name:` line is the Nth row (index N-1) in the role
checklist screen. If a host needs roles at indices 1, 5, 10, 12, 14
(ascending), the sequence from a checklist that starts at cursor 0 is:

```
DOWN 1   SPACE   # 0 -> 1
DOWN 4   SPACE   # 1 -> 5
DOWN 5   SPACE   # 5 -> 10
DOWN 2   SPACE   # 10 -> 12
DOWN 2   SPACE   # 12 -> 14
ENTER            # submit the checklist
```

Each `DOWN <n>` is the *delta* from the previous stop, not from 0 —
this is the most common transcription error.

## 3. The off-by-one that actually happened: vault entry count

A vault skeleton generated by `pilot inventory generate` had 8 scalar
entries:

```
ipa_admin_password
grafana_admin_password
restic_aws_access_key_id
restic_aws_secret_access_key
restic_password
thanos_aws_access_key_id
thanos_aws_secret_access_key
alertmanager_config          <- not intended to be edited
```

The script only intended to edit the first 7 and skip
`alertmanager_config`. Because the "save & exit" item's index was
computed as if there were only 7 entries (`DOWN 8` to reach "💾 存檔並
離開"), it actually landed one row early — on "➕ 新增 key" — since the
real list is `[entry0..entry7 (8 items), 新增key(8), 存檔並離開(9),
不存檔離開(10)]`.

**The fix is not "count the entries you plan to edit" — it's "count
every entry currently in the file, including ones you're leaving
untouched."** The generated file's own item list is ground truth; the
runbook's or your own summary's list of "entries I care about" is not.

## 4. When landing on the wrong item silently derails the rest

Once the cursor lands on the wrong menu item (here: "➕ 新增 key"
instead of "💾 存檔並離開"), the *next* scripted characters get typed
into whatever prompt that wrong item opens — in this case a "new key
name" text field. Subsequent `DOWN`/`ENTER` events sent as if for later
steps get consumed as garbage keystrokes/newlines into that field,
producing a loop of validation errors ("不能留空") until the script
runs out. The process still exits 0 when `trec drive`'s own `--timeout`
elapses or the script ends — **it does not fail loudly**. The tell is
the *absence* of the expected success line (`✅ 已存檔 <path>`) in the
output, not a non-zero exit code.

Always `grep` the resulting log/transcript for the wizard's own
success confirmation before trusting a `trec drive` run did what you
intended.

## 5. Re-verify item counts after every `pilot inventory generate`

`pilot inventory generate` can add group_vars files and vault keys you
didn't explicitly ask for (backfilled from `.example.yml` for whatever
roles are actually in use). Re-run the `grep`/read-the-file step from
§3 *after* generation, immediately before writing the vault-fill
script — don't reuse an item count from before generation ran.
